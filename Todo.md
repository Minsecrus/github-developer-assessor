# GitHub 用户能力评估工具设计说明

## 一、核心问题：评分应衡量绝对能力还是相对能力？

在设计 GitHub 用户能力评估体系时，首先需要明确评分目标：**评分究竟是在衡量开发者的绝对能力水平，还是衡量其在开发者群体中的相对位置。**

这一选择会直接影响评分模型、数据采集方式以及最终结果解释。

### 1. 绝对能力评估（Absolute Assessment）

绝对能力评估关注的是：

> “该用户是否达到某个能力标准？”

它不关心用户之间的竞争关系，而是判断其是否具备某种确定能力。

典型案例：

* 驾照考试：核心目标是判断考生是否具备安全驾驶能力，并不存在“只允许前10%通过”的机制。
* 工程能力认证：例如判断一个开发者是否具备独立完成某类项目的能力。
* 科研问题解决能力：如解决某个数学问题、完成某项技术突破。

### 绝对能力评估的典型特征：

#### 1. 不存在固定人数限制

通过人数理论上可以无限增长。

例如：

* 驾考不会因为今年报名人数增加而降低通过人数。
* 如果所有开发者都达到某个工程能力标准，都应该获得对应评价。

#### 2. 难度变化会直接影响整体结果

当评估标准发生变化时，所有人的结果会同步变化。

例如：

* 自动挡考试推广后，驾考难度降低，通过率提高。
* 某项技术工具普及后，完成任务的整体门槛降低。

因此，绝对能力评分更接近：

> “能力标准 → 用户是否达到标准”

---

### 2. 相对能力评估（Relative Assessment）

相对能力评估关注的是：

> “该用户相比其他用户处于什么位置？”

它本质上是一种竞争排序。

典型案例：

* 高考
* 公务员考试
* 各类竞赛
* 奥林匹克数学竞赛

这些体系评估的不只是“会不会”，而是：

> “相比其他参与者，你表现得更好吗？”

### 相对能力评估的典型特征：

#### 1. 存在竞争性名额限制

例如：

* 公务员岗位只录取固定人数。
* 竞赛排名决定获奖者范围。

#### 2. 难度变化不会显著改变排名结构

例如：

* 某一年高考题目变难，整体分数下降，但进入顶尖高校的人数比例基本保持稳定。
* 竞赛题目难度变化，主要影响整体得分，而不会改变核心竞争逻辑。

因此，相对能力评分更接近：

> “用户表现 → 群体排序”

---

# 二、不同评分模式对 GitHub 能力评估的影响

## 1. 如果采用绝对能力评分

优势：

* 结果容易解释。
* 可以直接对应工程能力等级。

例如：

* 3分：能够完成基础项目开发。
* 4分：能够独立完成中型项目。
* 5分：能够设计复杂系统并维护大型工程。

这种评分方式能够回答：

> “这个开发者是否具备某种能力？”

类似工程能力认证。

---

但其主要问题是：

## 容易产生评分集中化（Score Compression）

随着 AI Agent 等工具的发展，越来越多开发者可以快速生成：

* README 文档
* 基础代码结构
* 测试代码
* 项目说明

导致部分指标出现整体提升。

例如：

如果仓库文档质量满分为 5 分：

* 传统情况下，用户可能分布在 2~5 分。
* AI 辅助后，大量用户可能集中在 4.0~4.5 分。

此时：

* 4.2 分和 4.4 分之间可能不存在明显能力差异。
* 但如果直接用于排名，会产生大量细微但缺乏实际意义的差距。

因此，绝对评分更适合作为：

> 能力认证，而不是竞争排序。

---

## 2. 如果采用相对能力评分

优势：

能够解决不同开发环境、工具变化导致的评分膨胀问题。

例如：

随着 AI 编程工具普及，所有人的代码生成能力提升，那么评估重点可以转向：

* 问题解决能力
* 长期维护能力
* 创新能力

通过与大量样本比较，判断用户在开发者群体中的位置。

类似：

* 百分位排名（Percentile）
* Elo 类评分
* 标准化评分

例如：

> “该用户 GitHub 综合能力超过 85%的开发者。”

---

潜在实现方式：

可以建立开发者样本库：

1. 选取大量具有代表性的 GitHub 用户。
2. 根据多个维度建立评分模型：

   * 项目规模
   * 技术复杂度
   * 代码质量
   * Issue/PR 活跃度
   * 长期维护能力
   * 社区影响力
3. 计算目标用户在样本中的相对位置。

---

# 三、仓库选择问题：历史项目不一定代表当前能力

目前评分体系会纳入用户历史仓库，但 GitHub 仓库存在一个重要问题：

> GitHub 记录的是开发者的全部历史轨迹，而不是当前能力快照。

其中可能包含：

* 学习阶段项目
* 教程跟随项目
* 实验性项目
* 已废弃项目
* 多年前低水平作品

这些仓库可能会降低评估准确性。

因此需要引入：

## 仓库权重机制（Repository Weighting）

可以综合以下因素：

### 1. 时间衰减（Time Decay）

近期项目权重更高。

例如：

[
W_t=e^{-\lambda \Delta t}
]

其中：

* (\Delta t)：距离当前时间
* (\lambda)：衰减系数

避免多年以前的项目持续影响当前评价。

---

### 2. 活跃度权重

例如：

优先考虑：

* 最近一定时间内 commit 数量较高的仓库
* 最近仍有维护行为的项目

避免：

> 一个五年前的大型仓库，仅因为代码量巨大而成为主要评价对象。

---

### 3. 项目规模与质量综合权重

不能简单按照代码量排序。

因为：

* 代码量大 ≠ 技术复杂
* 历史屎山项目可能拥有巨大规模

可以综合：

* 有效代码量
* 提交频率
* Issue 活跃度
* PR 情况
* 技术栈复杂度
* 架构复杂度

建立仓库代表性评分：

[
RepositoryScore=f(Size, Complexity, Activity, Quality, Recency)
]

---

# 四、建议方向

GitHub 用户能力评估不应该单纯采用“考试式分数”，而应该拆分为两个维度：

## 1. 绝对能力评分

回答：

> “这个开发者能够完成什么？”

例如：

* 是否能够独立开发 Web 应用。
* 是否能够维护中型项目。
* 是否具备系统设计能力。

## 2. 相对能力评分

回答：

> “这个开发者在开发者群体中的位置。”

例如：

* 超过多少比例的开发者。
* 属于初级、中级、高级开发者中的哪一层。

最终结果可以采用：

> 能力等级 + 百分位排名

的双指标形式。

例如：

> 工程能力等级：高级
> GitHub 综合能力：超过 87%的开发者

这样既保留绝对能力的可解释性，又具备相对评价的区分能力。
